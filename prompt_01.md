# Build Prompt: Text-Based Football Management Simulation (Phase 1 — Engine & League Core)

You are acting as a software engineer building a deep football (soccer) management simulation, in the spirit of Football Manager, using a real player attribute dataset. This is a **multi-phase project**. This prompt defines **Phase 1 only**: a working simulation engine and a single-league season simulation. Do not attempt transfers, scouting, contracts, press conferences, multi-league structures, or graphical polish yet — those come later. Depth should go into the *simulation logic*, not the presentation layer.

## 1. Objective

Build a locally-runnable simulation where:
- Real players (from the dataset described below) are grouped into real clubs.
- A single league of clubs plays a full round-robin season (or a season using a subset of clubs if the full dataset is too large to start with).
- Match results are generated by an attribute-driven simulation engine — not random/coinflip results, but results that emerge from player attributes, team strength, and in-match events.
- The user can run a season and see: fixtures, results, a live/scrolling match event feed (text commentary), and a league table that updates correctly.

**Success criterion for Phase 1:** I can start the program, simulate a full season, and watch a league table build up match by match, with results that are clearly influenced by squad quality (i.e., a team of 17-rated youth players should not consistently beat a team of 15-rated senior professionals — verify this statistically before calling it done).

## 2. Hard Constraints

- **No complicated installs.** No game engines (Unity/Godot/etc.), no heavy graphics stacks, no requiring the user to install Node build tooling, Docker, or a database server. Prefer either:
  - **Python 3 standard library + `sqlite3`**, with output either as clean terminal text or a single local webpage served via Python's built-in `http.server`/lightweight framework (one `pip install` at most, e.g. Flask), OR
  - **A single self-contained HTML file with vanilla JS** that loads pre-processed JSON/CSV client-side and runs entirely in the browser with no backend at all.
  Pick one approach and justify it briefly before building. Either way: one command (or one double-click) to run.
- **No graphics/sprites needed.** "Visuals" means clean, readable text/tables/a scrolling event log — not animations or pitch graphics. Think: an old-school text match engine (score, minute, event log, league table), styled simply if it's HTML.
- **Deep simulation over deep UI.** Prioritize simulation realism (attribute interactions, variance, injuries/fatigue eventually) over how it looks.
- **Performance matters.** The full dataset is 90,000+ players. Don't do expensive per-attribute computation across the whole database on every tick — only simulate in detail for the ~18–24 players actually involved in a given match (starting XIs + subs). Load/query efficiently (SQLite with indexes, not repeated full CSV scans).

## 3. The Dataset

You are being given a real player database exported from a Football Manager-style scouting tool. It comes as one or more CSV files (this prompt includes a small sample — the real files, ~90,000+ rows total, will be provided separately/attached, possibly split into multiple part-files that must be merged and de-duplicated by `UID`).

### 3.2 Attribute block (all 1–20 scale)

**Technical:** `Cor` corners, `Cro` crossing, `Dri` dribbling, `Fin` finishing, `Fir` first touch, `Fre` free kicks, `Hea` heading, `Lon` long shots, `L Th` long throws, `Mar` marking, `Pas` passing, `Pen` penalties, `Tck` tackling, `Tec` technique

**Mental:** `Agg` aggression, `Ant` anticipation, `Bra` bravery, `Cmp` composure, `Cnt` concentration, `Dec` decisions, `Det` determination, `Fla` flair, `Ldr` leadership, `OtB` off the ball, `Pos` positioning, `Tea` teamwork, `Vis` vision, `Wor` work rate

**Physical:** `Acc` acceleration, `Agi` agility, `Bal` balance, `Jum` jumping reach, `Nat.1` natural fitness (⚠️ column name collides with `Nat` = nationality — confirm by header index, not name, when parsing), `Pac` pace, `Sta` stamina, `Str` strength

**Goalkeeping:** `Aer` aerial reach, `Cmd` command of area, `Com` communication, `Ecc` eccentricity, `Han` handling, `Kic` kicking, `1v1` one-on-ones, `Pun` punching, `Ref` reflexes, `TRO` tendency to rush out, `Thr` throwing

**Hidden/personality:** `Ada` adaptability, `Amb` ambition, `Cons` consistency, `Dirt` dirtiness, `Imp M` important matches, `Inj Pr` injury proneness, `Loy` loyalty, `Pres` pressure, `Prof` professionalism, `Spor` sportsmanship, `Temp` temperament, `Vers` versatility, `Media Handling` (categorical personality tags, e.g. "Media-friendly", "Volatile, Confrontational")

### 3.4 Data reality checks to build in

- Column `Nat.1` is a naming collision (nationality vs. Natural Fitness) — parse by **position index**, never by header name lookup alone, or rename on ingestion.
- Multi-position strings need a real parser (`D/WB/AM (R)` = can play Defender, Wing-Back, and Attacking Mid, all on the right side).
- Height/weight/foot-strength are categorical or unit-suffixed strings, not raw numbers — normalize all of them during ingestion into a clean numeric schema (e.g., an SQLite table with proper types) before the engine ever touches them.
- Expect the full dataset to arrive as multiple CSV parts; write an ingestion script that merges and de-dupes by `UID`, and is idempotent (safe to re-run).
- Don't assume every quirk above is complete — when the real files arrive, run a schema/quirk audit (unique values per categorical column, null rates, outliers) before building the engine on top of it.

## 4. Phase 1 Scope — build this, nothing more

**In scope:**
1. Data ingestion: CSV(s) → clean SQLite database (players, clubs, derived attributes).
2. A club/league structure: group players by `Club`, group clubs into one league (use `Based` or another grouping column once confirmed) with a sensible number of clubs (e.g., 16–20) for a first playable season — don't try to load all 90k players' clubs into one giant league.
3. A team-strength/lineup model: pick a plausible starting XI per club based on positions and attribute quality (simple "best XI for formation" logic is fine here — no human tactical AI yet).
4. **The match engine** (see §5) — this is the heart of the project and deserves the most design effort.
5. Season/fixture management: round-robin fixture generation, results processing, standings table (points, W/D/L, GF/GA, GD), sorted correctly.
6. A minimal but clean output: run a match and see a text event feed + final score; run/simulate a full season and see the table update.

**Explicitly out of scope for Phase 1** (note these for the roadmap, don't build them yet): transfers, contracts, training/attribute development over time, scouting/staff, finances, press interactions, multiple divisions/promotion-relegation, save/load across sessions beyond basic persistence, any 2D/graphical pitch view.

## 5. Match Engine Design Spec

Build a simulation that turns two lineups of attribute-rated players into a realistic scoreline and event log — not a random-number scoreline. A reasonable approach:

1. **Possession/phase model:** Simulate the match as a sequence of discrete "possessions" or attacking phases (not literal minute-by-minute physics). Each phase resolves through a chain of attribute duels.
2. **Duel resolution:** Model contests as opposed-attribute checks with randomness, e.g.:
   - Dribble vs. tackle: attacker's `Dri`/`Agi`/`Bal`/`Fla` vs. defender's `Tck`/`Ant`/`Pos`/`Str`.
   - Pass success: passer's `Pas`/`Tec`/`Vis`/`Cmp` vs. pressure from nearest defender's `Ant`/`Pos`.
   - Shot quality → goal: shooter's `Fin`/`Tec`/`Cmp`/`Lon` (for long shots) modulated by shot location/pressure, vs. keeper's `Ref`/`1v1`/`Pos`/`Han` and defenders' block chance (`Tck`/`Bra`/`Str` for blocks, `Hea`/`Jum` for aerial situations).
   - Aerial duels: `Hea`/`Jum`/`Str`/`Bra` vs. equivalent.
3. **Team-level modifiers:** formation/mentality (even a simple attacking/balanced/defensive toggle) should shift how many phases become "attacking" for each side and how risky the duels are.
4. **Variance & fatigue:** use `Sta` to degrade physical attributes over the match, `Cons`/`Temp`/`Pres`/`Imp M` to add performance variance, `Inj Pr` to occasionally trigger minor in-match events (optional stretch goal, not required for "done").
5. **Output:** the engine should emit a structured list of timestamped events (chance created, shot, save, goal, card, sub) that both the text log renderer and the final scoreline are derived from — don't compute a scoreline separately from the event log, or they'll disagree.
6. **Sanity testing:** before calling the engine done, run a batch (e.g., 100+ simulated matches) between a strong sample squad and a weak sample squad and confirm the strong squad wins convincingly more often, and that a closely-matched pair produces a believable distribution of results (mostly close games, occasional upsets) — share this validation output as evidence of "deep, not random."
